### 왜 리액트 인가?

리액트가 인기를 얻는 이유는 무엇일까?

- 명시적 상태 변경
  - 리액트는 단방향 바인딩만 지원한다. (말 그대로 데이터의 흐름이 한쪽으로만 간다는 것)
  - 리액트의 상태 변화는 ‘단방향’으로, 그리고 ‘명시적’으로 이뤄진다. 상태가 변화했다면 그 상태 변화를 명시적으로 일으키는 함수만 찾으면 된다. 이러한 리액트의 명시적인 상태 업데이트는 많은 개발자들에게 간단함과 유연함을 제공한다.
  - 하지만 단방향 바인딩은 항상 변화를 감지하고 업데이트하는 코드를 매번 작성해야하며 코드가 증가하는 단점이 있다.
- JSX(JavaScript XML)
  - 리액트는 JSX라는 문법을 사용한다. JSX는 자바스크립트의 확장 문법으로, 자바스크립트 코드 안에 마크업 코드를 작성할 수 있게 해준다. 이는 가독성을 높이고, 컴포넌트의 구조를 더 명확하게 표현할 수 있게 해준다.
- 비교적 배우기 쉽고 간결함
  - HTML과 자바스크립트를 어느정도 알고 있다면 배우기 쉽다.
- 강력한 커뮤니티 그리고 메타

### 리액트의 역사

- 2000년대의 웹 생태계
  - [LAMP 스택](https://www.ibm.com/kr-ko/topics/lamp-stack)이 주를 이뤘던 웹 개발 패턴은 서버에서 데이터를 받아와 HTML을 생성하고, 클라이언트에 전달하는 방식이었다. (콘텐츠를 서버에서 동적으로 생성하고 웹 브라우저는 이를 단순이 다운로드받아 렌더링하며 자바스크립트는 폼 처리같은 부수적인 역할만 하는 방식)

- 브라우저의 역할 확대
  - 시간이 지나면서 점차 다양한 기능을 브라우저에서 지원하기 시작했고, 서버뿐 아니라 클라이언트에서도 서버와 통신해 데이터를 불러오기 시작했다.
  - MVVM, MVC 패턴을 기반으로 나날이 복잡해져가는 자바스크립트 코드의 체계화를 위해 여러 라이브러리와 프레임워크가 등장했다.

- 리액트의 등장
  - 당시 페이스북은 많은 사용자가 이용하는 서비스인만큼 성능이 무엇보다 중요했다. 또한 실시간이 변경되는 화면을 빠르게 업데이트할 수 있는 기술이 필요했다.
  - 코드의 양이 늘어날수록 브라우저의 부담이 커져갔다. 페이스북은 여러 라이브러리를 사용해 구축한 프레임워크인 BoltJS를 개발했지만, 점점 더 복잡해지는 프로젝트 아키텍쳐의 문제로 인해 한계를 느꼈다.
  - 이후에 모델이 뷰를 변경하는 단방향 방식으로, 모델의 데이터가 변경되어 뷰가 변경되어야 하면 이전 DOM을 버리고 새롭게 렌더링하는 방식을 제안했다.

### 리액트를 사용하는 이유

1. 명시적 상태 변경
  - 리액트는 단방향 바인딩만 지원 (=데이터 흐름이 한쪽으로만)
  - 상태가 변화하면 그 상태 변화를 명시적으로 일으키는 함수만 찾으면 됨
  - 다만, 항상 변화를 감지하고 업데이트하는 코드를 매번 작성해야 한다
2. JSX
3. 비교적 배우기 쉽고 간결함
4. 강력한 커뮤니티 그리고 메타

# 리액트의 역사

- 2000년대 웹개발은 데이터베이스에서 필요한 데이터를 불러온다음, 웹 서버에서 HTML 페이지를 만들어 클라이언트에 제공하는 방식으로 작동
- 2010년대 자바스크립트가 자리잡고 ES5가 처음으로 표준 스펙이됨(DOM 수정, 인터렉션, Ajax 등) ⇒ 점점 코드가 복잡해짐
  - 체계화하고자 MVC, MVVM 패턴 나옴
  - 자바스크립트의 코드 크기가 늘어날 수록 → 브라우저에서 다운로드, 파싱, 실행하는 부담 커짐
- 이후에 Bolt.js, FBolt.js 등 많은 라이브러리들이 나왔지만 결국 단방향 바인딩이 되는 렌더링 방식을 제안
  - 모델이 뷰를 변경한다
  - 리액트 등장
- 리액트의 관심사 분리
  - HTML, JS, CSS가 각자 다른 파일에 있는 것이 관심사의 분리이지만
  - HTML, JS, CSS를 한 파일에 컴포넌트 단위로 분리하는 것도 관심사의 분리이다.
